# In-use dependency licenses

Snapshot of every package in the dependency tree (runtime + dev) and its license SPDX identifier. Used to verify AGPL-3.0-or-later compatibility.

**Regenerate with:**

```bash
pnpm licenses list --json | jq -r '
  to_entries[] | .key as $lic | .value[] |
  .name as $n | (.versions // [.version]) | .[] | "\($n)@\(.): \($lic)"
' | sort -u
```

We use `pnpm licenses list` rather than `license-checker --json` for this full-tree snapshot: under pnpm's default strict (non-hoisted) `node_modules` layout, `license-checker` walks only the top-level symlinked **direct** dependencies (~37 packages) and cannot traverse the content-addressable store, so it under-reports the transitive tree. `pnpm licenses list` reads the resolved lockfile and reports every package. Note it groups by package name, so a package resolved at multiple versions is split into one `name@version` line each by the `jq` above; it also omits the root project, so the `notebook-app@0.0.0: UNLICENSED` line is added manually. The blocking `license-check` CI gate still uses `license-checker --production` (production deps are direct + their resolved subtree, which `license-checker` does see correctly).

**Keep this up to date.** Re-run after any `pnpm install`, `pnpm update`, or dependency removal, and replace the list below. If a new license appears that isn't on the AGPL-compatible list, add the offending package to [incompatible.md](incompatible.md) and suggest removing or replacing it.

## AGPL-compatible license allowlist

If a future install introduces a license **not** on this list, treat it as a blocker until reviewed:

- MIT, ISC, BSD-2-Clause, BSD-3-Clause, 0BSD, BlueOak-1.0.0
- Apache-2.0
- MPL-2.0 (file-level copyleft, compatible)
- Python-2.0, Zlib, Unlicense
- CC0-1.0, CC-BY-4.0 (for non-code assets)
- WTFPL (and any `X OR Y` disjunction where at least one option is on this list)
- GPL-3.0-or-later, GPL-3.0-only, LGPL-3.0-or-later, AGPL-3.0-or-later, AGPL-3.0-only

The `-only` variants of GPL/AGPL are admitted because our project is itself AGPL-3.0-or-later and we don't need downstream relicensing flexibility. The `-or-later` form is still preferred for new deps because it leaves future relicensing options open.

Watch out for: custom/source-available licenses, CC-BY-NC-_, CC-BY-SA-_ on code, and the GPL-2.0 trap below. See [incompatible.md](incompatible.md) for traps already encountered.

### Bare `BSD` (no version)

`license-checker` infers `BSD` (no clause-count suffix) for packages where the `package.json` `license` field is missing or non-SPDX, and the LICENSE file's heuristic match lands on the BSD family. Currently in our tree this applies only to `parse-cache-control@1.0.1`, whose LICENSE file is the standard BSD-2-Clause text. The bare `BSD` allow-list entry exists to admit this case without forcing every BSD-inferred package onto an override list. If a future package resolves to bare `BSD` and inspection shows it's not actually BSD-family, that's a blocker — investigate the LICENSE file.

## Keeping the workflow allow list in sync

This allowlist is enforced in CI by the `license-check` job in `.github/workflows/ci.yml`, which passes a `--onlyAllow` string to `license-checker`. **The two lists must stay in sync.** When evaluating a new license:

1. Read the upstream LICENSE file (not just the `license` field) and confirm AGPL-3.0-or-later compatibility against the considerations in this document.
2. If approved: add the SPDX identifier to **both** the list above and the `--onlyAllow` string in `ci.yml`. Re-run `license-checker` per the regen command above and update the package list.
3. If rejected: add an entry to [incompatible.md](incompatible.md) with the package name, the license, and the reasoning. Do **not** add to the allowlist.
4. The workflow uses _exact-string match_ on license identifiers — SPDX expressions like `(MIT OR CC0-1.0)` must be added as literal strings if they appear in our tree.

### The GPL-2.0 trap

Package metadata frequently labels a dependency `GPL-2.0` without disambiguating `-only` vs `-or-later`. The distinction matters:

- **GPL-2.0-or-later** is fine. Downstream consumers can relicense to GPL-3.0, which is AGPL-3.0-compatible (GPLv3 §13 added the bridge).
- **GPL-2.0-only** is a blocker when _linked_ into our binary. GPL-2.0 requires the combined work be distributed under GPL-2.0, and it doesn't grant permission to add AGPL-3.0 §13's network-use clause — the two licenses impose conflicting requirements that can't both be satisfied.

**IPC caveat.** "Linked" is the load-bearing word. If we ship GPL-2.0-only code as a separate process talking to us over IPC (pipes, sockets, HTTP, the Python-sidecar pattern we already use for faster-whisper / Docling / Supertonic), the FSF's position is that the two programs are not a single combined work and the license conflict doesn't trigger — each process can be distributed under its own license. This is the same reasoning that lets us ship AGPL-licensed sidecars at all.

So: a GPL-2.0-only **npm dependency** (linked into our JS bundle) is a blocker. A GPL-2.0-only **standalone binary** invoked as a subprocess or sidecar is likely fine, but treat it as a per-case legal call rather than an allowlist entry — document the reasoning in [incompatible.md](incompatible.md) when accepting one.

When a dep is labeled bare `GPL-2.0`, check the upstream `LICENSE` file to see if it actually says "version 2 or, at your option, any later version" before treating it as a blocker.
